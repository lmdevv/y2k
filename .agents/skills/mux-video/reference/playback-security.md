# Playback Security (Signed URLs and DRM)

Comprehensive guide to securing video playback with signed URLs, JSON Web Tokens (JWT), playback restrictions, and Digital Rights Management (DRM) protection including Widevine, FairPlay, and PlayReady.

## Understanding Playback Policies

When you upload a video or initiate a live stream through Mux, you can define what type of access should apply to your content by specifying a playback policy.

Mux offers two kinds of playback policies:

- **Public**: Enables playback URLs that can be watched anywhere, at any time, without any restrictions
- **Signed**: Enables playback URLs that require a valid JSON Web Token (JWT) to gain access. The JWT should be signed and generated by your application on a protected server, not on a public client

A playback policy can be specified when you create a new asset or live stream, or can be added to an existing asset or live stream. Each asset or live stream can have multiple playback IDs with different policies.

## Signed Playback Policies

When you apply a signed playback policy to your content, there are two ways to restrict video playback:

1. **Expiration time** (required): Specify when the issued JWT should expire. All HTTP requests past the expiration time are denied.
2. **Playback Restrictions** (optional): Implement additional rules such as Referrer Validation and User-Agent Validation.

### Creating an Asset with Signed Playback

```json
// POST https://api.mux.com/video/assets
{
  "inputs": [
    {
      "url": "https://storage.googleapis.com/muxdemofiles/mux-video-intro.mp4"
    }
  ],
  "playback_policies": [
    "signed"
  ],
  "video_quality": "basic"
}
```

## Signing Keys

Signing keys are used to generate valid JWTs for accessing your content. When you create a new signing key, the API generates a 2048-bit RSA key-pair and returns the private key and a generated key-id.

**Important**: Mux signing keys are different than Mux API keys.

You should securely store the private key for signing tokens, while Mux stores the public key to validate the signed tokens.

### Creating a Signing Key

```json
// POST https://api.mux.com/system/v1/signing-keys

// Response:
{
  "data": {
    "private_key": "(base64-encoded PEM file with private key)",
    "id": "(unique signing-key identifier)",
    "created_at": "(UNIX Epoch seconds)"
  }
}
```

Signing keys are created and deleted independently of assets. You typically need only one signing key active at a time, but you can create multiple to enable key rotation.

## Playback Restrictions

Mux supports two types of playback restriction:

- **Referrer Validation**: Restricts whether the domain in the HTTP `Referer` header will be allowed for playback
- **User Agent Validation**: Restricts whether high-risk user agents or missing user agents will be allowed for playback

### Creating a Playback Restriction

```json
// POST https://api.mux.com/video/v1/playback-restrictions
{
  "referrer": {
    "allowed_domains": [
      "example.com"
    ],
    "allow_no_referrer": false
  }
}
```

### Response

```json
{
  "data": {
    "updated_at": "1634595679",
    "referrer": {
      "allowed_domains": [
        "example.com"
      ]
    },
    "id": "JL88SKXTr7r2t9tovH7SoYS8iLBVsjZ2qTuFS8NGAQY",
    "created_at": "1634595679"
  }
}
```

### Playback Restriction Syntax

You can specify both referrer and user agent restrictions in the same request:

```json
{
  "referrer": {
    "allowed_domains": [
      "*.example.com",
      "foo.com"
    ],
    "allow_no_referrer": false
  },
  "user_agent": {
    "allow_no_user_agent": false,
    "allow_high_risk_user_agent": false
  }
}
```

**Domain Syntax Options**:

| Pattern | Behavior |
|---------|----------|
| `[]` (empty array) | Deny all domains |
| `["*.example.com", "example.com"]` | Allow example.com and all subdomains |
| `["*"]` | Allow any domain |
| `["*.example.com"]` | Allow one level of subdomains (denies xyz.foo.example.com) |

### Playback Restriction Limits

- Up to 100 different Playback Restrictions per environment
- Up to 100 different domains per Playback Restriction
- Playback Restriction ID can be used for a single video or a group of videos

### Referrer Header Considerations

- Web browsers send the website address in the `Referer` HTTP header
- Native iOS and Android applications do not send this header
- First-party Apple devices (like Apple TV 4K) never set a referrer header
- If using Chromecast, add `www.gstatic.com` to allowed domains
- If using AirPlay, add `mediaservices.cdn-apple.com` and set `allow_no_referrer` to true for first-party Apple devices

**Recommendation**: Create two Playback Restriction objects - one with `allow_no_referrer` set to `true` for native apps, and one set to `false` for web applications.

### User-Agent Validation

- `allow_no_user_agent`: If false, denies requests without a User-Agent value
- `allow_high_risk_user_agent`: Mux maintains a list of user agents associated with higher risk playback

## JSON Web Token (JWT) Structure

### Required Claims

| Claim | Description | Value |
|-------|-------------|-------|
| `sub` | Subject of the JWT | Mux Video Playback ID |
| `aud` | Audience (intended application) | `v` (Video/Subtitles), `t` (Thumbnail), `g` (GIF), `s` (Storyboard), `d` (DRM License) |
| `exp` | Expiration time | UNIX Epoch seconds when token expires |
| `kid` | Key Identifier | Key ID from signing key creation |

### Optional Claims

| Claim | Description |
|-------|-------------|
| `playback_restriction_id` | Playback Restriction Identifier for validation |

### Expiration Time Guidelines

- Set expiration to at least the duration of the Asset or expected Live Stream duration
- When the signed URL expires, playback stops even if already started
- Handle cases where users leave and return to your application - fetch new signed URLs as needed

## Signing the JWT

Sign JWTs using the RS256 algorithm with your private key.

### Using the Mux Node SDK

```js
const mux = new Mux({
  tokenId: "your-access-token-id",
  tokenSecret: "your-access-token-secret",
  jwtSigningKey: "your-environment-signing-public-key",
  jwtPrivateKey: "your-environment-signing-private-key"
});

const playbackToken = await mux.jwt.signPlaybackId("your-playback-id", {expiration: '7d'});
const drmLicenseToken = await mux.jwt.signDrmLicense("your-playback-id", {expiration: '7d'});
```

## Using Signed URLs

Supply the JWT in the resource URL using the `token` query parameter:

**Video URL**:
```
https://stream.mux.com/{playback-id}.m3u8?token={JWT}
```

**Thumbnail URL**:
```
https://image.mux.com/{playback-id}/thumbnail.{format}?token={JWT}
```

**Important**: Passing `token` for public playback IDs will fail. This is intentional to prevent false appearance of security.

### Query Parameters and Signing

When signing a URL, parameters must be included in the JWT claims, not in the URL:

**Correct** Signed URL:
```
https://image.mux.com/{signed_playback_id}/thumbnail.jpg?token={token}
```

**Incorrect** Signed URL:
```
https://image.mux.com/{signed_playback_id}/thumbnail.jpg?time=25&token={token}
```

Include parameters in the claims body:

```json
{
  "sub": "{PLAYBACK_ID}",
  "aud": "{AUDIENCE_TYPE}",
  "exp": "{EXPIRATION_TIME}",
  "redundant_streams": true
}
```

### Custom Parameters for Tracking

You can pass custom parameters via a `custom` key for tracking purposes (e.g., identifying bad actors who share URLs):

```json
{
  "sub": "{PLAYBACK_ID}",
  "aud": "{AUDIENCE_TYPE}",
  "exp": "{EXPIRATION_TIME}",
  "custom": {
    "session_id": "xxxx-123"
  }
}
```

**Guidelines**:
- Do NOT include personally identifiable information (PII)
- Nest custom parameters inside the `"custom"` key

## Digital Rights Management (DRM)

DRM provides an extra layer of content security for video content streamed from Mux, blocking or limiting:

- Screen recording
- Screen sharing
- Downloading tools

Mux uses industry standard protocols: Google Widevine, Microsoft PlayReady, and Apple FairPlay.

### How Mux DRM Protects Content

Protection comes in three parts:

1. **Video encryption**: Prevents playback without proper license
2. **Screen capture protection**: Prevents screenshots and screen recording
3. **HDCP**: Prevents recording from video outputs like HDMI

### Device Protection Levels

| Device Type | Encrypted Video | Screen Capture Protection | HDCP Enforced |
|-------------|-----------------|---------------------------|---------------|
| iPhone/iPad | Yes | Yes | Yes |
| Modern Android devices | Yes | Usually | No |
| Older/lower-end Android | Yes | Sometimes | No |
| Chrome/Edge on desktop | Yes | Sometimes | No |

### DRM Prerequisites

1. **Request a FairPlay certificate** from Apple for playback on Apple devices
2. **Request DRM access** via Settings -> Digital Rights Management in the Mux Dashboard
3. **Receive your DRM configuration ID** after Mux enables DRM on your environment

### FairPlay Certificate Request

Requirements:
- Apple Developer account with active subscription
- Team account owner access (if part of a team)

Key form responses:
- "Do you have a third-party streaming distribution partner?" - Select "Yes"
- "Streaming Distribution Partner Name" - Enter "Mux, Inc."
- "Streaming Distribution Partner Website" - Enter "https://mux.com"

**Note**: You do not need to request Widevine or PlayReady certificates - Mux manages these automatically.

### Creating DRM Protected Assets

Use `advanced_playback_policies` with your DRM configuration ID. Video quality must be `plus` or `premium`:

```json
// POST /video/v1/assets
{
  "inputs": [
    {
      "url": "https://storage.googleapis.com/muxdemofiles/mux.mp4"
    }
  ],
  "advanced_playback_policies": [
    {
      "policy": "drm",
      "drm_configuration_id": "your-drm-configuration-id"
    }
  ],
  "video_quality": "plus"
}
```

**Important**: You cannot use both `playback_policy` and `advanced_playback_policies` in the same request.

### Creating DRM Protected Live Streams

```json
// POST /video/v1/live-streams
{
  "advanced_playback_policies": [
    {
      "policy": "drm",
      "drm_configuration_id": "your-drm-configuration-id"
    }
  ],
  "new_asset_settings": {
    "advanced_playback_policies": [
      {
        "policy": "drm",
        "drm_configuration_id": "your-drm-configuration-id"
      }
    ]
  }
}
```

### DRM Playback Tokens

For DRM playback, you need two tokens:
1. **Playback token**: For video stream access
2. **DRM license token**: For decryption license

```js
const playbackToken = await mux.jwt.signPlaybackId("your-playback-id", {expiration: '7d'});
const drmLicenseToken = await mux.jwt.signDrmLicense("your-playback-id", {expiration: '7d'});
```

### DRM with Mux Web Player

```html
<mux-player
  playback-id="your-playback-id"
  playback-token="your-playback-token"
  drm-token="your-drm-license-token"
></mux-player>
```

Support for DRM in Mux Player was added in version 2.8.0.

### DRM with Mux Player iOS

```swift
let playbackOptions = PlaybackOptions(
  drmToken: "your-drm-license-token",
  playbackToken: "your-playback-token",
)

let playerItem = AVPlayerItem(
  playbackID: "your-playback-id",
  playbackOptions: playbackOptions
)
```

Support for DRM in Mux Player for iOS was added in version 1.1.0.

### DRM with Mux Player Android


### Testing DRM

To verify DRM is working: take a screenshot. If DRM is active, the video will appear as either a black rectangle or a single frame from the start of the video.

### DRM Security Levels

Mux's DRM defaults to a balance of security and playability, automatically leveraging higher security levels where available. Contact Mux support to adjust security levels for specific requirements (e.g., Hollywood studio security requirements).

**Note**: Only video tracks are DRM protected. Audio-only assets and live streams are not protected by DRM.
