# Mux Data Metrics Overview

Mux Data provides comprehensive metrics to measure video engagement and Quality of Experience (QoE). This reference covers all available metrics including engagement metrics, QoE scores, and real-time monitoring metrics.

## Views and Watch Time

### Definition of a View

In Mux Data, a "View" is an attempt (successful or not) to play a video. A view is created when:
- A viewer clicks play
- Playback is started programmatically

Key behaviors:
- If a user taps play, the video starts loading and fails, that counts as a single view
- If a user taps play, starts watching, pauses, then resumes within 60 minutes, that counts as a single view
- Each view is tracked until playback is explicitly ended or 60 minutes after playback stops
- A single video can be watched multiple times in a single view by looping or seeking to the start

View lifecycle:
- After a view stops receiving playback events for 60 seconds, it is considered complete and available in Metrics
- Playback events will be added to views if playback resumes within 60 minutes
- After 60 minutes of inactivity, the view is finalized and new playback events create a new view

### How Watch Time is Calculated

Watch Time is the cumulative amount of time the user spent watching or attempting to watch the video. This includes:
- Actively playing content
- Starting up
- Rebuffering
- Seeking

**Example:** If a user watches for 90 seconds, has 4 seconds of rebuffering, spends 2 seconds seeking by rewinding, then watches 60 more seconds, the total Watch Time would be 156 seconds (90 + 4 + 2 + 60).

**Note:** If a user watches a 2-minute video at 2x speed, Watch Time will be 1 minute (assuming no buffering, seeking, or startup time). Watch Time measures elapsed time during playback, not video duration consumed.

## Metric Aggregation

Engagement and Quality of Experience metrics are tracked during each playback attempt and a value is assigned to each generated view. Reports aggregate values from individual, completed views matching specified filters.

- **Engagement metrics:** Use the start time of each view for time range filtering
- **QoE metrics:** Aggregated based on the end time of each view
- Views are only included in metric calculations if they have a valid value for that metric

## Engagement Metrics

### Views

Views counts the total number of views that started during the selected time interval. This is calculated differently from QoE metrics, which count views that ended during the interval.

|----------|-------|
| Type | Number |
| Measurement | Count |

Counts distinct viewers based on the start time of the associated view, using Viewer ID to determine uniqueness.

- If no Viewer ID is provided, the default Viewer ID generated by the Mux SDK is used
- Recommended: Set a meaningful, anonymized Viewer ID for accurate counts
- The Viewer ID should not contain personally identifiable information (email, username, etc.)

| Property | Value |

### Ad Attempts

Counts the number of times an ad attempted to start playing.

- Ad Attempts occur when each individual ad is attempted during an ad break

|----------|-------|

### Ad Breaks

Counts the number of times ad breaks occurred.

| Property | Value |

## Quality of Experience (QoE) Metrics

QoE metrics describe the actual end-user experience of watching video, as opposed to QoS (Quality of Service) metrics which describe system performance without reference to user experience.

### Overall Viewer Experience Score

A high-level score from 0-100 that measures the overall Quality of Experience. A score of 100 means every viewer had a satisfying experience; a score of 0 means every viewer had a frustrating experience.

The score is based on four other Viewer Experience Scores:
- Playback Success
- Startup Time
- Smoothness (Playback Smoothness)
- Video Quality

#### Formula

Each individual video view receives an experience score of 0-100. The Overall Viewer Experience Score is calculated by averaging experiences across all views.

The formula uses weighted tradeoff scores based on user research:

```
Playback Success Score * (T_Sm,Q + T_Sm,Su + T_Su,Q) / 3
```

Where:
- T_Sm,Q = Tradeoff(Smoothness, Quality)
- T_Sm,Su = Tradeoff(Smoothness, Startup)
- T_Su,Q = Tradeoff(Startup, Quality)

**Relative importance (from most to least):**
1. Playback Success
2. Smoothness
3. Startup Time
4. Quality

| Property | Value |
|----------|-------|
| Type | Score (0-100) |
| Measurement | Average |
| Filterable | Yes |

**Use this metric to:**
- See trends in overall viewer experience over time
- Prioritize improvement efforts in areas with the lowest scores

## Playback Success Metrics

### Playback Success Score

Focuses on whether video played back successfully. Considers two components:
- Did the video play without an error?
- Did the user actually get to playback, or did they exit before it started?

#### Scoring

| Scenario | Score |
|----------|-------|
| Successful playback | 100 |
| Exit before video start (EBVS) | 50 |
| Playback failure | 0 |
| EBVS < 1 second | N/A (excluded) |

**Why EBVS = 50?** While exits can indicate streaming problems (e.g., video took too long to load), some exits are normal (user clicked wrong video, saw a different video to watch). Views abandoned in less than one second are assumed unintentional or programmatic and excluded.

| Property | Value |
|----------|-------|
| Type | Score (0-100) |
| Measurement | Average |
| Filterable | Yes |

### Exits Before Video Start

Percentage of video views where the viewer clicked play but the video never began playing (Video Startup Time was never recorded), excluding playback failures.

**Use this metric to:**
- Watch how changes in Video Startup Time impact viewer abandonment
- Compare players and understand if factors other than startup time cause viewers to leave (loading indicators, poster frames)

| Property | Value |
|----------|-------|
| Type | Percentage |
| Measurement | Average |

### Playback Failure Percentage

Percentage of video views that failed due to a fatal error. Playback failures can happen at any point, causing playback to end prematurely.

An error is a Playback Failure if:
- It has a severity of fatal
- It is not due to a business rule exception

| Property | Value |
|----------|-------|
| Type | Percentage |
| Measurement | Average |

### Video Startup Failure Percentage

Percentage of video views that experienced an error preventing the user from seeing the first frame of video (ads or content).

An error is a Startup Failure if:
- It occurs before the first frame of video is shown
- It has a severity of fatal
- It is not due to a business rule exception

| Property | Value |
|----------|-------|
| Type | Percentage |
| Measurement | Average |

### Business Exception Percentage

Percentage of video views that failed due to a fatal business rule exception. Business Rule Exceptions can happen at any point during playback.

| Property | Value |
|----------|-------|
| Type | Percentage |
| Measurement | Average |

### Video Startup Business Exception Percentage

Percentage of video views that experienced a business rule exception preventing the user from seeing the first frame of video.

| Property | Value |
|----------|-------|
| Type | Percentage |
| Measurement | Average |

### View Dropped Percentage

Video views sometimes end for unknown reasons:
- Application or player crash
- Loss of internet connection
- Browser behavior restricting analytics

This metric counts views where the session ended without a clean exit.

| Property | Value |
|----------|-------|

## Startup Time Metrics

### Startup Time Score

Describes viewer satisfaction with startup time. Longer startup times mean lower scores.

#### Formula

```
(8 / (8 + startup_time_in_seconds)) * 100
```

The score decreases at a greater rate after 500ms of starting up. Once startup time reaches around 8 seconds, the rate of score decay decreases since additional seconds become less impactful.

**Note:** EBVS views do not receive a Startup Time Score.

#### Example Values

| Startup Time | Score |
|--------------|-------|
| 400 ms | 95 |
| 2 seconds | 80 |
| 8 seconds | 50 |
| 20 seconds | 29 |

| Property | Value |
|----------|-------|
| Type | Score (0-100) |
| Measurement | Average |
| Filterable | Yes |

### Video Startup Time

Time from when the player is instructed to play (user click or autoplay) to when the first frame of video is showing and the playhead is progressing. For players with pre-roll ads, this is time until the first frame of the pre-roll ad.

**Percentiles provided:**
- **Median (50th):** Typical experience (half are better, half are worse)
- **95th Percentile:** Poorer experience, excluding outliers (1 in 20 views)

**Impact factors:**
- Network performance
- Initial rendition selection
- Preloading (only recommended when video is primary content)

| Property | Value |
|----------|-------|
| Type | Milliseconds |
| Measurements | Median, 95th Percentile |
| Filterable | Yes |

### Player Startup Time

Time from when the player is first initialized to when it is ready to receive instructions.

**Note:** Requires passing `player_init_time` in your client integration.

| Property | Value |
|----------|-------|
| Type | Milliseconds |
| Measurements | Median, 95th Percentile |
| Filterable | Yes |

### Page Load Time

Time from initial user request for a page to when the video player is first initialized. Only recorded for the first video view on a page.

| Property | Value |
|----------|-------|
| Type | Milliseconds |
| Measurements | Median, 95th Percentile |
| Filterable | Yes |

### Aggregate Startup Time

Combines Page Load Time, Player Startup Time, and Video Startup Time to show total wait time after requesting to watch a video.

Only recorded for the first video view on a page.

| Property | Value |
|----------|-------|
| Type | Milliseconds |
| Measurements | Median, 95th Percentile |
| Filterable | Yes |

### Seek Latency

Average time viewers wait for video to start playing after seeking to a new time.

Seeking includes:
- Viewer clicking on progress bar
- Programmatic jumps (e.g., resuming where previously stopped)

If user pauses during seek, still measures time until video is ready to resume.

| Property | Value |
|----------|-------|

## Smoothness Metrics

### Smoothness Score

Measures rebuffering impact. Higher scores mean less rebuffering.

#### Formula

Average of two components:

**Component 1 (Rebuffer Count):**
```
(1 / sqrt(1 + (rebuffer_count / 2)^2)) * 100
```

**Component 2 (Rebuffer Percentage):**
```
e^(-10 * rebuffer_percentage) * 100
```

**Why two components?**
- Multiple short interruptions can be worse than a single longer one
- Rebuffering as a percentage matters (5 seconds in a 10-second clip vs. 2-hour movie)
- Averaging balances cases where one metric is high while the other is low

**Note:** EBVS views do not receive a Smoothness Score.

#### Examples

| Scenario | Score |
|----------|-------|
| No rebuffering | 100 |
| 5-minute video with single 5s rebuffer | 85 (avg of 80 and 90) |
| 20-minute video with four 15s rebuffers | 54 (avg of 44 and 60) |

| Property | Value |
|----------|-------|
| Type | Score (0-100) |
| Measurement | Average |
| Filterable | Yes |

### Rebuffer Percentage

Rebuffer Duration as a percentage of Watch Time.

Rebuffering occurs when video stalls while attempting to play, usually because download time exceeds playback time.

| Property | Value |
|----------|-------|
| Type | Percentage |
| Measurement | Average |
| Relative To | Watch Time |
| Filterable | Yes |

### Rebuffer Frequency

How often rebuffering events happen (per minute of watch time). Reveals video stuttering issues where the player restarts too aggressively.

| Property | Value |
|----------|-------|

| Measurement | Average |
| Relative To | Watch Time |
| Filterable | Yes |

### Rebuffer Duration

Amount of time (in seconds) viewers wait for rebuffering per video view.

| Property | Value |

### Rebuffer Count

Number of rebuffering events per video view.

## Video Quality Metrics

### Video Quality Score

Measures visual quality by comparing video stream resolution to player resolution. Significant upscaling degrades quality and viewer experience.

#### Formula

```
e^(-0.33 * (0.15 * U_max + 0.85 * U_avg)) * 100
```

Where:
- U_max = Max Upscale Percentage
- U_avg = Average Upscale Percentage

85% of the score is based on average upscaling; 15% on max (peak) upscaling.

**Note:** EBVS views do not receive a Video Quality Score.

#### Examples

| Scenario | Score |
|----------|-------|
| No upscaling | 100 |
| 50% upscaling throughout | 85 |
| 200% upscaling for 30s, then none for 20 min | 92 |

| Property | Value |
|----------|-------|
| Type | Score (0-100) |
| Measurement | Average |
| Filterable | Yes |

### Upscale Percentage

Measured when the video player increases video size to fill the display.

**Calculation:** Change in one dimension (the dimension that fits first when upscaling).

**Example:** 320x240 source to 640x480 player = 100% upscale ((640-320) / 320).

Time-weighted: If upscaled 100% for half the video and 0% for half, Total Upscale Percentage = 50%.

| Property | Value |
|----------|-------|
| Type | Percentage |
| Measurements | Average, 95th Percentile, Median |
| Filterable | Yes |

### Downscale Percentage

When video source is too big and must be reduced. Does not reduce quality but wastes bandwidth.

| Property | Value |
|----------|-------|
| Type | Percentage |
| Measurements | Average, 95th Percentile, Median |
| Filterable | Yes |

### Max Upscale/Downscale Percentage

Reveals points of significant scaling, even if brief. Helps identify which rendition may be the culprit.

### Weighted Average Bitrate

Time-weighted average of indicated bitrates experienced during a video stream.

**Example:** 1 minute at 1Mbps + 2 minutes at 2Mbps = 1.67 Mbps ((1*1 + 2*2) / 3).

Only includes bitrates of segments actually played, not downloaded-but-discarded segments.

| Property | Value |
|----------|-------|
| Type | Mbps |
| Measurements | Median, 95th Percentile |
| Filterable | Yes |

### Live Stream Latency

Time from camera capture to viewer seeing the action (ingest-to-render latency).

Uses `EXT-X-PROGRAM-DATE-TIME` tags in HLS manifests compared to current UTC time.

**Typical latencies:**
- Standard HLS: ~30 seconds
- Low Latency HLS (LL-HLS): 5-10 seconds

**Notes:**
- Viewers playing >5 minutes behind live edge are excluded
- Mux Video inserts PDT tags at ingest time, so latency may be ~1 second lower than actual glass-to-glass

| Property | Value |
|----------|-------|
| Type | Milliseconds |
| Measurements | Median, 95th Percentile |
| Filterable | Yes |

### Request Throughput

Average throughput (Mbps) for completed media requests.

**Formula:**
```
sum(bits downloaded) / sum(request time)
```

**Example:** Three 40Mb downloads in 4s, 2s, 2s = 15 Mbps ((40+40+40) / (4+2+2)).

| Property | Value |
|----------|-------|
| Type | Mbps |
| Measurements | Median, 95th Percentile |
| Filterable | Yes |

### Request Latency

Average time to first byte for media requests.

**Formula:**
```
sum(time to first byte) / (number of requests)
```

| Property | Value |
|----------|-------|
| Type | Milliseconds |
| Measurements | Median, 95th Percentile |
| Filterable | Yes |

### Max Request Latency

Maximum time to first byte for any individual media request.

| Property | Value |
|----------|-------|
| Type | Milliseconds |
| Measurements | Median, 95th Percentile |
| Filterable | Yes |

## Real-Time Monitoring Metrics

Mux Data Monitoring offers near real-time metrics (sub-20 second latency) available for 24 hours. Available on Media plans.

### Current Concurrent Viewers (CCV)

Number of viewers currently watching a video.

**Includes:**
- Viewers waiting for video to start
- Viewers experiencing rebuffering
- Viewers who just experienced playback failure

**Excludes:**
- Paused viewers
- Viewers rebuffering for more than 5 consecutive minutes

### Video Startup Failures by Start Attempts

Percentage of viewers experiencing a video startup failure (error preventing first frame of video) as a percent of Start Attempts.

**Start Attempts:** Viewers in the video loading state or who just transitioned to startup success, startup failure, or exit before video starts.

### Playback Failures by CCV

Percentage of viewers who experienced a playback failure (fatal error preventing future playback) as a percent of CCV.

**Note:** Non-fatal errors are not included. Measured differently from Playback Failure Percentage in QoE Metrics.

### Exits Before Video Start by Start Attempts

Percentage of viewers who abandoned while waiting at least one second for video to start, as a percent of Start Attempts.

**Note:** Measured differently from Exits Before Video Start in QoE Metrics.

### Current Rebuffering Percentage

Time viewers spent rebuffering from last measurement to current time, as a percentage of total watch time.

**Note:** Measured differently from Rebuffer Percentage in QoE Metrics.

### Current Average Bitrate

Average of video bitrates shown to viewers over the time period, based on manifest-indicated rendition values.

### Video Startup Time (Monitoring)

Current median startup time - half of viewers experience faster, half experience slower.
